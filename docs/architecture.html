Overview

Key Design Decisions

1. Partitioning Strategy

Weather fact table is partitioned by month on observed_at. This keeps index sizes small and enables efficient time-based queries through partition pruning.

2. Idempotent Operations

All transforms use upsert logic with unique constraints. This allows safe reruns without data duplication and handles partial failures gracefully.

3. Append-Only Raw Layer

Raw tables store full JSON payloads and never update or delete. This provides complete audit trail and enables reprocessing with different logic.

4. Type-2 Slowly Changing Dimensions

Wikipedia page dimension uses type-2 SCD to track title changes over time. Current version marked with is_current = true and valid_to = NULL.

5. Concurrent Materialized Views

Mart views refresh using REFRESH MATERIALIZED VIEW CONCURRENTLY. This allows non-blocking reads during refresh operations.

6. Fail-Fast Data Quality

Data quality checkpoints run after transforms, before mart refresh. If checkpoints fail, flow stops and marts are not refreshed, preventing bad data from reaching analytics.

Data Flow

1. Extract: Fetch data from APIs (Open-Meteo, Wikipedia) ‚Üì 2. Load: Store full JSON payloads in raw tables ‚Üì 3. Transform: Parse JSON, explode arrays, convert units ‚Üì 4. Quality: Run Great Expectations checkpoints ‚Üì 5. Core: Upsert into fact and dimension tables ‚Üì 6. Mart: Refresh materialized views (concurrent) ‚Üì 7. Analytics: Query mart layer for reporting

Performance Optimizations

  • Monthly Partitioning: Keeps indexes small, enables partition pruning
  • Local Indexes: Index on (location_id, observed_at) per partition
  • Concurrency Control: Max 3 concurrent API calls to respect rate limits
  • Concurrent Refresh: Materialized views refresh without blocking reads
  • Future Scaling: Citus sharding by location_id when needed (local, no cloud cost)
